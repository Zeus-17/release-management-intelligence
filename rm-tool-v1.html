📖 What is a Defect? What is a Bug? — Definitions & Classification Guide
▾ Show
Defect
Any deviation from expected or specified behaviour. The broadest category — includes bugs, missing requirements, incorrect data, performance degradation, and usability failures. A defect can exist in code, configuration, documentation, data or process.
Bug
A specific type of defect in software code — a programming error that causes incorrect or unexpected behaviour. All bugs are defects, but not all defects are bugs. A missing feature, misconfigured environment or flawed test data is a defect, not a bug.
Regression
A defect in functionality that previously worked correctly — broken by a new change. Regression testing specifically looks for these. A regression defect always traces back to a specific change in the current or recent release.
Release Blocker
A defect that must be resolved before go-live. Criteria: prevents a core user journey; causes data loss or corruption; violates regulatory compliance; or would require emergency patching within 48 hours of go-live. Cannot be deferred without Director-level written waiver.
Severity vs Priority: Severity = how badly the system is broken (technical impact). Priority = how urgently it must be fixed (business urgency). A cosmetic defect on the CEO's dashboard may be Low Severity but Critical Priority. Always assess both — they drive different decisions.
Release Blocker criteria (this tool): Defects classified as Blockers prevent go-live. The Go/No-Go readiness indicator turns RED until all Blockers are resolved or formally waived. Deferred defects require approved waiver documentation.

🔗 URL Parameter Integration
The simplest integration path. Any system that can construct a URL (a Slack bot, a Jira automation, a pipeline step) can open this tool with fields pre-filled. No server, no API key required.
http://your-host/rm-tool-v1.html ?release=Q2+Platform+Release &type=major &date=2026-07-01 &rm=Jane+Smith &teams=Platform%2C+QA%2C+DevOps &dm=cloud,api,mobile release — Release name / ID type — major | minor | patch | emergency | pilot | feature date — ISO date YYYY-MM-DD rm — Release Manager name teams — Comma-separated team list dm — Delivery models (comma-separated) notes — URL-encoded scope notes scenario — Practice scenario ID (1–6)

ℹ️ What is a PIR? — click to expand definition
▾
PIR — Post Implementation Review
A formal evaluation conducted after a release has gone live and the hypercare period has ended. It captures what was delivered vs what was planned, what worked and what didn't, improvement actions, and produces a signed document for governance and audit purposes.

Also known as: After Action Review (AAR) · Post Mortem · Release Review · Lessons Learned · Implementation Review
Your organisation may use a different name. You can rename this throughout the tool in .
